## 1. 개념 정의: 이벤트 전파의 3단계

브라우저에서 이벤트가 발생하면 단순히 그 지점에서 끝나는 것이 아니라, DOM 트리를 따라 세 단계를 거치며 이동한다.

1.  **캡처링 단계 (Capturing Phase)**: 이벤트가 `window`에서 시작해 타겟 요소까지 **내려가는** 단계이다.
2.  **타겟 단계 (Target Phase)**: 이벤트가 실제 발생한 요소에 **도달**한 단계이다.
3.  **버블링 단계 (Bubbling Phase)**: 이벤트가 다시 타겟 요소에서 최상위 요소로 **올라가는** 단계이다.

---

## 2. 비교 분석: 버블링 vs 캡처링

| 구분 | 이벤트 버블링 (Bubbling) | 이벤트 캡처링 (Capturing) |
| :--- | :--- | :--- |
| **방향** | 타겟 → 부모 → `window` (상향식) | `window` → 부모 → 타겟 (하향식) |
| **기본값** | 브라우저 기본 동작 (활성화) | 기본적으로 비활성화 |
| **설정 방법** | `addEventListener(type, handler)` | `addEventListener(type, handler, true)` |
| **주요 활용** | **이벤트 위임(Event Delegation)** | 로그 수집, 하위 요소 클릭 사전 차단 |

---

## 3. 심화 개념 보강: 제어 및 최적화

### **① 이벤트 전파 제어 메서드**
* **`stopPropagation()`**: 현재 이벤트가 더 이상 상위(버블링)나 하위(캡처링)로 퍼지지 않도록 즉시 중단시킨다.
    * *주의*: 전파를 막으면 분석 도구(GA 등)가 클릭을 감지하지 못하는 부작용이 생길 수 있으므로 신중히 사용해야 한다.
* **`stopImmediatePropagation()`**: 현재 요소에 등록된 **다른** 리스너들까지도 실행되지 않게 막는다.
* **`preventDefault()`**: 전파와는 상관없이, 브라우저의 **기본 행동**(링크 이동, 폼 제출 등)만 취소한다.

### **② 이벤트 위임(Event Delegation)의 이점**
버블링을 이용한 이벤트 위임은 프론트엔드 성능 최적화의 핵심이다.
* **메모리 절약**: 수천 개의 리스트 항목(`li`)에 각각 리스너를 다는 대신, 부모(`ul`) 하나에만 달아 메모리 사용량을 대폭 줄인다.
* **동적 요소 대응**: 자식 요소가 새로 추가되거나 삭제되어도 부모 리스너가 그대로 동작하므로 추가적인 코드 작성이 필요 없다.
