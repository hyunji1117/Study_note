# SSR (Server Side Rendering) 정리 (Frontend 관점)

## 1. SSR과 CSR의 개념 차이

웹 애플리케이션에서 화면을 그리는 방식은 크게 두 가지로 나뉜다.

### **SSR (Server Side Rendering)**

* **개념**: 서버에서 완전한 형태의 HTML을 생성하여 클라이언트에 전달하는 방식이다.
* **동작**: 클라이언트는 전달받은 HTML을 파싱하여 즉시 화면에 그린다.

### **CSR (Client Side Rendering)**

* **개념**: 브라우저가 빈 뼈대 HTML과 자바스크립트 번들을 받아와 클라이언트 측에서 동적으로 컨텐츠를 채우는 방식이다.
* **동작**: 자바스크립트가 실행되기 전까지 사용자는 빈 화면을 보게 된다.

---

## 2. Next.js에서의 SSR과 Hydration

Next.js는 SSR의 장점을 극대화하기 위해 **Hydration** 개념을 사용한다.

1. **Pre-rendering**: 서버에서 정적인 HTML을 먼저 내려주어 사용자가 빠르게 화면을 볼 수 있게 한다.
2. **Hydration**: 내려받은 HTML에 자바스크립트 이벤트 리스너를 부착하여, 정적인 페이지를 상호작용이 가능한 동적 페이지로 전환하는 과정을 거친다.

---

## 3. SSR의 장점

* **SEO(검색 엔진 최적화) 유리**: 크롤러가 완성된 HTML 컨텐츠를 쉽게 수집할 수 있어 검색 결과 상단 노출에 유리하다.
* **빠른 초기 로딩 속도(FCP)**: 자바스크립트 번들을 모두 다운로드하고 실행할 때까지 기다리지 않아도 되므로 사용자가 체감하는 초기 화면 표시 속도가 빠르다.
* **저사양 기기 배려**: 클라이언트의 연산 부담이 CSR보다 적다.

---

## 4. SSR의 단점

* **TTV와 TTI의 간극**: 화면은 보이지만(Time To View) 실제 클릭 등 상호작용이 안 되는(Time To Interactive) 시간 차이가 발생하여 사용자에게 답답함을 줄 수 있다.
* **서버 부하 증가**: 요청 시마다 서버에서 페이지를 생성해야 하므로 WAS 서버의 리소스 비용이 증가한다.
* **사용자 경험(UX) 제약**: 전통적인 SSR은 페이지 이동 시 화면 깜빡임이 발생하며, 클라이언트 사이드 라우팅에 비해 매끄러움이 떨어진다.
* **높은 구현 복잡도**: 서버와 클라이언트 환경을 모두 고려하여 로직을 작성해야 하므로 설계가 까다롭다.

---

## 5. 요약 및 비교

| 구분 | SSR (Server Side Rendering) | CSR (Client Side Rendering) |
| --- | --- | --- |
| **렌더링 주체** | 서버 (Server) | 브라우저 (Client) |
| **초기 로딩 속도** | 빠름 | 느림 |
| **SEO** | 매우 유리 | 불리 (보완 필요) |
| **서버 자원** | 많이 사용함 | 적게 사용함 |
| **주요 용도** | 블로그, 커머스, 뉴스 등 | 관리자 페이지, 대시보드 등 |
