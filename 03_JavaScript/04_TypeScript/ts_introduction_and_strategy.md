## 1. 개념 정의

### **타입스크립트(TypeScript)란?**
자바스크립트에 **정적 타입 체킹** 시스템을 추가한 오픈 소스 프로그래밍 언어이다. 자바스크립트의 모든 기능을 포함하면서도(Superset), 컴파일 단계에서 오류를 잡아내어 런타임 안정성을 극대화한다. 

---

## 2. 비교 분석: JavaScript vs TypeScript

| 구분 | JavaScript (JS) | TypeScript (TS) |
| :--- | :--- | :--- |
| **타입 결정 시점** | **동적 타이핑**: 실행 시(Runtime) 결정 | **정적 타이핑**: 컴파일 시(Compile-time) 결정 |
| **에러 발견** | 코드를 실행해야 오류 확인 가능 | 코드 작성 중 IDE에서 즉시 확인 가능 |
| **도구 지원** | 기본적인 자동완성 제공 | 강력한 인텔리센스 및 리팩토링 지원 |
| **러닝 커브** | 낮음 (배우기 쉬움) | 보통 (타입 시스템 학습 필요) |
| **프로젝트 규모** | 소규모, 빠른 프로토타이핑에 유리 | 대규모, 장기 유지보수 프로젝트에 필수 |

---

## 3. 심화 개념 보강: 도입 전략과 주의사항

### **① 점진적 채택(Gradual Adoption)**
TS는 한 번에 모든 코드를 바꿀 필요가 없다. 
* **`allowJs: true`**: 기존 JS 파일을 유지하면서 새로운 기능만 TS로 작성할 수 있다.
* **`any` 타입의 지양**: 도입 초기에는 편리할 수 있지만, 남용하면 TS의 이점이 사라지므로 점진적으로 구체적인 인터페이스(Interface)를 정의해 나가는 것이 중요하다.

### **② 개발자 경험(DX)과 문서화**
TS의 타입 정의는 그 자체로 **살아있는 문서** 역할을 한다. 
* 별도의 문서를 보지 않아도 함수가 어떤 인자를 받고 어떤 값을 반환하는지 IDE에서 즉시 확인할 수 있다.
* 이는 협업 시 커뮤니케이션 비용을 획기적으로 줄여주는 실질적인 이점이 된다.

### **③ 도입이 망설여지는 상황 (Trade-off)**
* **초기 설정 비용**: 환경 설정(tsconfig 등)과 타입 정의에 시간이 소요된다.
* **컴파일 단계**: 코드를 바로 실행할 수 없고 반드시 컴파일 과정을 거쳐야 한다.
