## 1. 개념 정의

### **HTTP (Hypertext Transfer Protocol)**
웹 브라우저(클라이언트)와 서버가 HTML, 이미지, JSON 등 다양한 데이터를 주고받기 위해 사용하는 **통신 규약**이다.

* **동작 방식**: 클라이언트가 **요청(Request)**을 보내면 서버가 **응답(Response)**하는 단방향 모델이다.
* **비상태성 (Stateless)**: 서버는 이전 요청의 상태를 기억하지 않으며, 각 요청은 독립적으로 처리된다.
* **HTTPS**: HTTP에 **TLS/SSL** 암호화 계층을 추가하여 데이터 전송의 보안을 강화한 버전이다.

---

## 2. RESTful API: 리소스를 다루는 약속

REST(Representational State Transfer)는 웹의 기존 HTTP 인프라를 최대한 활용하기 위한 **아키텍처 스타일**이다.

| 구성 요소 | 표현 방식 | 예시 |
| :--- | :--- | :--- |
| **자원 (Resource)** | **URI** (명사 형태) | `/users`, `/posts/123` |
| **행위 (Verb)** | **HTTP Method** (동사 형태) | `GET`, `POST`, `PUT`, `DELETE` |
| **표현 (Representation)** | **Payload** (데이터 포맷) | JSON, XML 등 |

---

## 3. 심화 개념 보강: REST의 4가지 핵심 원칙

작성하신 규칙들을 프론트엔드 개발 관점에서 재해석했다.

1.  **클라이언트-서버 분리**: UI(프론트엔드)와 데이터(백엔드)의 역할을 명확히 나누어 각각 독립적으로 진화할 수 있게 한다.
2.  **무상태성 (Stateless)**: 서버가 클라이언트의 컨텍스트(로그인 상태 등)를 유지하지 않으므로, 요청 시 필요한 모든 정보(토큰 등)를 헤더에 담아 보내야 한다.
3.  **일관된 인터페이스 (Uniform Interface)**: 리소스 식별과 조작 방식이 표준화되어 있어, API 명세서만 보고도 기능을 쉽게 예측할 수 있다.
4.  **캐시 가능성**: HTTP의 `Cache-Control` 헤더를 활용하여 동일한 요청에 대해 서버 부하를 줄이고 응답 속도를 높힌다.
