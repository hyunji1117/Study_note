## 1. 개념 정의

### **헤더 (Header)**
데이터를 전송할 때 본문 앞에 붙는 **메타 데이터(Meta-data)**이다. 데이터 자체보다는 **"이 데이터를 어떻게 해석하고 처리해야 하는가"**에 대한 가이드라인을 담고 있다.

### **바디 (Body / Payload)**
클라이언트와 서버가 실제로 주고받고자 하는 **실제 데이터(Payload)**이다. 서비스의 핵심 비즈니스 로직에 필요한 정보(사용자 정보, 게시글 내용 등)가 담기는 곳이다.

---

## 2. 비교 분석

| 구분 | 헤더 (Header) | 바디 (Body) |
| :--- | :--- | :--- |
| **역할** | 데이터에 대한 설명 및 제어 정보 (안내 역할) | 전송하고자 하는 본질적인 데이터 (본문) |
| **주요 내용** | 인증(Authorization), 타입(Content-Type), 캐시 제어 등 | JSON, HTML, Form Data, 이미지 파일 등 |
| **크기 제한** | 보통 8KB ~ 16KB (웹 서버 설정에 따름) | 헤더에 비해 훨씬 크며 설정에 따라 대용량 가능 |
| **데이터 형태** | Key-Value 쌍의 텍스트 형식 | 다양한 포맷 (JSON, XML, Binary 등) |

---

## 3. 심화 개념 보강

### **① 헤더 크기 제한과 413 에러**
HTTP 표준 규격상 헤더 크기에 명시적인 제한은 없으나, 실제로는 보안과 성능을 위해 서버(Nginx, Apache 등)에서 제한을 둔다. 
* **문제 발생**: 쿠키가 너무 많이 쌓이거나 인증 토큰(JWT)이 지나치게 길어지면 헤더 크기 제한을 초과할 수 있다.
* **결과**: 서버는 `413(Content Too Large)` 또는 서버 설정에 따라 `431(Request Header Fields Too Large)` 응답을 보낸다.

### **② 데이터 타입의 이정표: Content-Type**
바디에 담긴 데이터가 어떤 형식인지 헤더의 `Content-Type`을 통해 알려주어야 한다.
* 만약 바디에 JSON을 담아 보내면서 헤더에 `Content-Type: application/json`을 명시하지 않으면, 서버는 바디 데이터를 단순 문자열로 오해하여 파싱 에러를 일으킬 수 있다.

### **③ 보안과 헤더**
바디는 암호화하지 않으면 노출되기 쉽지만, 민감한 인증 정보는 주로 헤더의 `Authorization` 필드에 담아 관리한다. 다만, HTTPS를 사용하면 헤더와 바디 모두 암호화되어 전송되므로 보안 통신이 가능해진다.
