## 1. 개념 정의

### **리액트 컴포넌트(React Component)**
UI를 구성하는 **독립적이고 재사용 가능한 코드 조각**이다. 레고 블록처럼 각각의 컴포넌트를 조립하여 복잡한 웹 애플리케이션을 완성한다.

* **함수형 컴포넌트**: 현재 리액트의 표준으로, Hooks를 사용하여 상태와 생명주기를 관리하며 코드가 간결하다. 
* **클래스형 컴포넌트**: 과거에 주로 사용되었으나, 현재는 특수한 경우(예: Error Boundary)를 제외하고는 함수형으로 대체되는 추세이다.

---

## 2. 컴포넌트 설계의 핵심 원칙

| 원칙 | 내용 | 기대 효과 |
| :--- | :--- | :--- |
| **단일 책임 원칙** | 하나의 컴포넌트는 한 가지의 역할만 수행하도록 설계 | 코드 파악이 쉽고 디버깅이 용이함 |
| **비즈니스 로직 분리** | UI를 그리는 로직과 데이터를 처리하는 로직을 분리 | UI 변경 시 로직 수정 없이 유연하게 대응 가능 |
| **높은 재사용성** | Props를 통해 데이터와 동작을 주입받아 유연하게 설계 | 코드 중복을 줄이고 개발 속도를 향상시킴 |
| **성능 최적화** | 메모이제이션(memo, useMemo 등)을 통한 불필요한 리렌더링 방지 | 대규모 애플리케이션에서도 부드러운 사용자 경험 제공 |

## 3. 심화 개념 보강: "컴포넌트가 춤을 춘다"

유연한 컴포넌트는 상황에 따라 다양한 모습을 보여준다. 이를 위해 프론트엔드 생태계에서는 다음과 같은 패턴을 자주 사용한다.

* **Presentational & Container 패턴**: 데이터 페칭을 담당하는 컴포넌트(Container)와 화면 렌더링만 담당하는 컴포넌트(Presentational)를 나누어 관리한다.
* **Composition (합성)**: 상속 대신 컴포넌트 안에 다른 컴포넌트를 담는 방식으로 기능을 확장한다.
* **Custom Hooks**: 여러 컴포넌트에서 공통적으로 사용하는 비즈니스 로직을 별도로 추출하여 재사용성을 극대화한다.

> **💡 개발자의 태도**
> 정답인 설계는 없다. 프로젝트의 규모, 팀의 컨벤션, 그리고 서비스의 목적에 따라 **유지보수성, 성능, 재사용성 사이의 균형**을 맞추는 것이 실력 있는 개발자의 핵심 역량이다.
