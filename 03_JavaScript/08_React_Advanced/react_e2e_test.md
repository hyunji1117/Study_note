## 1. E2E(End-to-End) 테스트의 개념

E2E 테스트는 애플리케이션의 사용자 경험을 **처음부터 끝까지(End-to-End) 시뮬레이션**하여 시스템 전체가 의도대로 작동하는지 검증하는 방식이다.

* **동작 방식**: 실제 브라우저 환경에서 버튼 클릭, 데이터 입력, 페이지 이동 등 사용자의 동작을 그대로 재현한다.
* **주요 도구**: `Cypress`, `Playwright` 등이 대표적이다.

---

## 2. E2E 테스트 vs 유닛 테스트

안정적인 애플리케이션을 위해서는 각 테스트의 역할 분담이 중요하다.

### 유닛 테스트 (Unit Test)

* **범위**: 개별 함수, 컴포넌트 등 최소 단위의 코드.
* **장점**: 실행 속도가 매우 빠르고 디버깅이 용이함.
* **한계**: 각 조각은 정상이어도, 조각들이 합쳐졌을 때 발생하는 시스템적 오류(예: 로그인 후 세션 유지 실패)는 잡기 어려움.

### E2E 테스트 (End-to-End Test)

* **범위**: 실제 사용자 시나리오 전체 (UI, API 호출, DB 연동 등).
* **장점**: 사용자가 겪을 수 있는 치명적인 흐름(Happy Path)을 검증하여 배포 리스크를 줄임.
* **한계**: 유닛 테스트에 비해 실행 속도가 느리고 구현 복잡도가 높음.

---

## 3. E2E 테스트의 필요성 및 효과

1. **사용자 관점의 검증**: 실제 사용 환경과 동일한 조건에서 테스트하므로, 사용자에게 직접적인 영향을 미치는 UI/UX 오류를 조기에 발견할 수 있다.
2. **시스템 통합 확인**: 프론트엔드뿐만 아니라 백엔드 API와의 통신, 화면 전환 등 모든 구성 요소가 함께 작동하는 과정을 점검한다.
3. **상호 보완적 전략**: 유닛 테스트로 개별 로직의 정확성을 확보하고, E2E 테스트로 비즈니스 로직의 연속성을 보장하여 프로덕트의 신뢰성을 극대화한다.

---

## 4. 정리 및 비교

| 구분 | 유닛 테스트 | E2E 테스트 |
| --- | --- | --- |
| **테스트 대상** | 함수, 컴포넌트 로직 | 사용자 시나리오 (전체 시스템) |
| **실행 속도** | 매우 빠름 | 상대적으로 느림 |
| **주요 목적** | 코드의 세부 정확성 검증 | 비즈니스 흐름 및 통합 안정성 확인 |
| **비용** | 낮음 | 높음 |

➡️ **중요한 사용자 흐름(결제, 회원가입 등)에는 E2E 테스트를, 세부 로직에는 유닛 테스트를 사용하는 전략이 권장됨**
