## 1. 개념 정의

### **Error Boundary란?**
하위 컴포넌트 트리 어디에서든 **자바스크립트 에러를 기록하며, 깨진 컴포넌트 트리 대신 폴백(Fallback) UI를 보여주는** React 컴포넌트이다.

* **동작 원리**: 클래스형 컴포넌트의 `getDerivedStateFromError()`(에러 발생 시 상태 업데이트)와 `componentDidCatch()`(에러 정보 로깅) 메서드를 사용한다.
* **특징**: React 컴포넌트 트리 중 에러가 발생한 특정 부분만 격리하여, 앱 전체가 중단되는 '화이트 스크린' 현상을 방지한다.

---

## 2. 왜 사용하는가? (필요성 및 장점)

| 구분 | 설명 |
| :--- | :--- |
| **앱 안정성** | 특정 컴포넌트의 에러가 전체 애플리케이션으로 전파되어 뻗는 것을 차단한다. |
| **사용자 경험(UX)** | 에러 발생 시 당황스러운 빈 화면 대신, "문제가 발생했습니다"와 같은 안내 문구나 재시도 버튼을 제공한다. |
| **선언적 에러 처리** | 에러가 발생했을 때 "어떻게(How)" 처리할지 명령적으로 작성하지 않고, "어떤(What)" UI를 보여줄지 컴포넌트 구조로 선언한다. |
| **관심사 분리** | 비즈니스 로직과 에러 처리 로직을 분리하여 코드의 가독성과 유지보수성을 높힌다. |

## 3. 심화 개념 보강

### **① "선언형 처리"가 유지보수에 주는 이점**
명령형 방식(try-catch)은 로직 곳곳에 에러 처리 코드가 침투하여 본래의 비즈니스 로직을 흐리게 만든다. 반면, Error Boundary를 통한 선언형 방식은 다음과 같은 이점을 준다.
* **직관적 구조**: HTML 구조처럼 에러 범위를 감싸기만 하면 되므로 에러 영향 범위를 한눈에 파악할 수 있다.
* **중앙 집중 관리**: 공통 에러 UI나 로깅 로직을 Error Boundary 하나에서 관리하므로 수정이 용이하다.

### **② 주의사항 (Error Boundary가 잡지 못하는 에러)**
모든 에러를 잡을 수 있는 것은 아니다. 다음 상황은 별도의 처리가 필요하다.
* **이벤트 핸들러**: 버튼 클릭 시 발생하는 에러 (일반 `try-catch` 사용 권장)
* **비동기적 코드**: `setTimeout`이나 `requestAnimationFrame` 내부 에러
* **서버 사이드 렌더링(SSR)**
* **Error Boundary 자체에서 발생하는 에러**
