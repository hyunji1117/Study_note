## 1. 개념 정의: 낙관적 업데이트란?

서버의 응답이 오기 전, **클라이언트의 UI를 먼저 성공 상태로 업데이트**하는 기법이다. 네트워크 지연(Latency)을 사용자에게 숨겨서 앱이 마치 즉각적으로 반응하는 것 같은 초고속 경험을 제공한다.

### **핵심 프로세스**

1. **이벤트 발생**: 사용자가 버튼 클릭 (예: 좋아요, 장바구니 담기)
2. **UI 선업데이트**: 서버 요청과 동시에 화면의 상태를 변경
3. **서버 요청**: 백엔드 API 호출
4. **결과 처리**:
* **성공 시**: 그대로 유지 (사용자는 이미 결과를 확인한 상태)
* **실패 시**: 이전 상태로 **롤백(Rollback)** 및 에러 핸들링

---

## 2. 비교 분석: 일반 업데이트 vs 낙관적 업데이트

| 구분 | 일반 업데이트 (Pessimistic) | 낙관적 업데이트 (Optimistic) |
| --- | --- | --- |
| **신뢰도** | 매우 높음 (확정된 데이터만 표시) | 보통 (실패 시 롤백 필요) |
| **반응 속도** | 서버 응답 속도에 의존 (느릴 수 있음) | **즉각적 (0ms)** |
| **구현 난이도** | 낮음 | 높음 (취소/롤백 로직 필수) |
| **사용자 경험** | 로딩 스피너/스켈레톤 노출 | 부드럽고 끊김 없는 상호작용 |
| **적합한 상황** | 결제, 송금, 회원가입 등 중요 데이터 | 좋아요, 댓글 달기, 할 일 체크, 북마크 |

---

## 3. 심화 개념 보강

### **① 롤백(Rollback)의 중요성**

낙관적 업데이트의 가장 큰 숙제는 '실패했을 때 어떻게 원래대로 돌리는가'이다.

* **스냅샷 저장**: 업데이트 직전의 UI 상태를 메모리에 임시 저장해 두어야 한다.
* **에러 알림**: 단순히 롤백만 하면 사용자가 당황할 수 있으므로, "네트워크 문제로 반영되지 않았습니다"와 같은 메시지를 띄우는 것이 좋다.

### **② 언제 쓰면 안 될까? (Anti-Patterns)**

사용자가 제출한 데이터가 서버에서 **가공되거나 검증(Validation)**을 거쳐야만 결과를 알 수 있는 경우에는 지양해야 한다.

* **예**: 중복 아이디 체크, 재고 확인이 필요한 결제, 복잡한 수식 계산 결과 등.

### **③ React Query(TanStack Query)에서의 구현**

실무에서는 직접 구현하기보다 라이브러리의 도움을 많이 받는다. `onMutate`에서 이전 값을 저장하고, `onError`에서 그 값을 다시 복구하는 패턴을 주로 사용한다.

```javascript
// React Query 예시 (의사 코드)
useMutation({
  mutationFn: updateTodo,
  onMutate: async (newTodo) => {
    await queryClient.cancelQueries(['todos']);
    const previousTodos = queryClient.getQueryData(['todos']); // 스냅샷 저장
    queryClient.setQueryData(['todos'], old => [...old, newTodo]); // 낙관적 업데이트
    return { previousTodos };
  },
  onError: (err, newTodo, context) => {
    queryClient.setQueryData(['todos'], context.previousTodos); // 실패 시 롤백
  },
});

```
