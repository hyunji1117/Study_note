## 1. 개념 정의

### **서버 컴포넌트 (Server Components)**
서버에서만 실행되고 렌더링되는 새로운 유형의 리액트 컴포넌트이다. 결과물로 HTML이 아닌 **특별한 형태의 JSON(RSC Payload)**을 브라우저에 전달하며, 브라우저는 이를 받아 화면을 재구성한다.

* **클라이언트 컴포넌트**: 기존처럼 브라우저에서 실행되며 `useState`, `useEffect` 등 인터랙션을 담당한다.
* **서버 컴포넌트**: 서버의 자원(DB, 파일 시스템)에 직접 접근하며 데이터 중심의 렌더링을 담당한다.

---

## 2. 장단점 비교 분석

| 구분 | 장점 | 단점 및 한계 |
| :--- | :--- | :--- |
| **성능** | 제로 번들 사이즈(Zero Bundle Size)로 초기 로딩 속도 향상 | 서버 응답 속도에 따른 렌더링 지연 발생 가능 |
| **보안** | DB 접근 로직이나 API 키 등 민감 정보가 노출되지 않음 | 클라이언트 상태(State)나 브라우저 API 사용 불가 |
| **데이터** | 서버에서 직접 데이터 페칭을 수행하여 네트워크 왕복 감소 | 정적 환경(Pure Static)에서의 활용 제약 |
| **UX** | 복잡한 로직을 서버에서 처리하여 저사양 기기 부담 완화 | 실시간 인터랙션(이벤트 핸들러 등) 구현의 복잡성 증가 |

---

## 3. 심화 개념 보강: 서버 컴포넌트의 제약 사항

서버 컴포넌트에서는 다음과 같은 기능을 **사용할 수 없다.**

* **Hooks 사용 불가**: `useState`, `useReducer`, `useEffect` 등 상태 및 생명주기 관리 불가능.
* **브라우저 API 접근 불가**: `window`, `document`, `localStorage` 등 브라우저 전용 객체 접근 불가능.
* **이벤트 핸들러**: `onClick`, `onChange`와 같은 사용자 상호작용 로직 작성 불가능.

> **💡 설계 전략: '서버'는 뼈대를, '클라이언트'는 근육을**
> 데이터가 필요한 레이아웃이나 리스트는 **서버 컴포넌트**로 구성하고, 검색창이나 좋아요 버튼처럼 즉각적인 반응이 필요한 부분만 **클라이언트 컴포넌트**로 쪼개어 감싸는 방식이 권장된다.
