## 1. 개념 정의

### **Streaming SSR이란?**
서버에서 전체 HTML이 완성될 때까지 기다리지 않고, **생성된 HTML 조각(Chunk)을 브라우저에 실시간 스트림 형태로 전달**하는 방식이다. 

* **기존 SSR**: `Data Fetching → HTML 렌더링 → 전송 → Hydration`의 전 과정을 모든 컴포넌트가 완료해야 화면이 보임.
* **Streaming SSR**: `Suspense`로 감싸진 무거운 컴포넌트를 제외한 'Shell(기본 틀)'을 먼저 보내고, 무거운 데이터는 준비되는 대로 후속 스트림으로 전달.

## 2. 비교 분석

| 구분 | 전통적 SSR (Legacy) | Streaming SSR (React 18+) |
| :--- | :--- | :--- |
| **전송 방식** | 전체 HTML 완성 후 일괄 전송 | 준비된 HTML 조각부터 점진적 전송 |
| **TTFB (첫 바이트 시간)** | 데이터 페칭 완료 시까지 지연됨 | **매우 빠름** (Shell 즉시 전송) |
| **사용자 경험** | 빈 화면 혹은 로딩바를 오래 봄 | 페이지 골격을 즉시 확인 가능 |
| **Hydration** | 페이지 전체가 한꺼번에 Hydration | **Selective Hydration** (준비된 곳부터) |
| **주요 API** | `renderToString` | `renderToPipeableStream` |

## 3. 심화 개념 보강: 불일치(Hydration Mismatch) 해결 전략

스트리밍 환경에서는 서버가 보낸 정적 HTML과 클라이언트 리액트가 렌더링한 결과가 일치해야 하는 **Hydration** 과정이 더욱 까다롭디.

### **① Selective Hydration (선택적 하이드레이션)**
React 18은 전체 페이지가 다 로드될 때까지 기다리지 않는다. 먼저 도착한 HTML 조각부터 인터랙션(클릭 등)이 가능하도록 부분적으로 연결한다. 만약 사용자가 아직 로드 중인 부분을 클릭하면, 리액트는 그 부분의 하이드레이션을 최우선 순위로 격상시킨다.

### **② 데이터 동기화와 Suspense**
* **동일한 데이터 소스**: 서버에서 렌더링할 때 사용한 데이터를 클라이언트도 그대로 물려받아야 합니다. (예: `React Query`의 `Hydrate` 방식)
* **상태 보존**: 서버와 클라이언트의 시간, 랜덤 값, 혹은 브라우저 전용 API(window 등) 사용으로 인한 차이는 `useEffect` 내부로 격리하여 불일치를 방지해야 한다.

### **③ SEO와 캐싱**
스트리밍된 HTML은 검색 엔진 크롤러가 읽을 수 있다. 다만, 일부 크롤러는 스트림이 끝날 때까지 기다리지 않을 수 있으므로, 핵심 메타 데이터는 반드시 첫 번째 스트림인 **'Shell'** 단계에 포함시켜야 한다.
