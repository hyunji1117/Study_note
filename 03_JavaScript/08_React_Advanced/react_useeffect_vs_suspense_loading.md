## 1. 개념 정의

### **useEffect + Local State (명령형 방식)**
컴포넌트 내부에서 `useState`로 로딩 상태(`isLoading`)를 직접 만들고, `useEffect` 안에서 비동기 통신 결과에 따라 이 상태를 수동으로 스위칭하는 방식이다. 

### **Suspense (선언형 방식)**
컴포넌트가 렌더링을 시도하다가 데이터가 없으면 렌더링을 '중단(Suspend)'하고, 가장 가까운 상위 `Suspense` 컴포넌트가 설정한 `fallback` UI를 대신 보여주는 방식이다. 

---

## 2. 비교 분석

| 구분 | useEffect + State | Suspense |
| :--- | :--- | :--- |
| **제어 방식** | **명령형**: 상태 변화를 직접 코드로 작성 | **선언형**: "로딩 중엔 이걸 보여줘"라고 정의 |
| **코드 구조** | 비즈니스 로직과 로딩 로직이 섞임 | 로딩 처리를 외부(부모)로 위임하여 깔끔함 |
| **데이터 읽기** | 렌더링 후(Post-render) 데이터 페칭 | 렌더링 중(Render-as-you-fetch) 데이터 읽기 |
| **UX 패턴** | 폭포수(Waterfall) 현상 발생 가능성 높음 | 여러 컴포넌트의 로딩 시점을 동기화하기 쉬움 |
| **구현 난이도** | 낮음 (표준 React 방식) | 보통 (지원 라이브러리나 특정 구조 필요) |

---

## 3. 심화 개념 보강

### **① 왜 Suspense는 '선언적'인가?**ㅈ
`useEffect` 방식은 "데이터를 가져오기 시작하면 로딩을 `true`로 바꾸고, 끝나면 `false`로 바꿔라"는 **절차**를 설명해야 합니다. 반면 `Suspense`는 "이 컴포넌트는 데이터가 올 때까지 기다릴 건데, 그동안은 이 스피너를 보여줘"라고 **목적**만 명시하면 나머지는 React 엔진이 알아서 처리한다.

### **② Suspense의 제약 사항**
* **Promise 연동 필수**: 일반적인 `fetch`는 `Suspense`를 트리거하지 못한다. `TanStack Query`나 `Next.js`처럼 `Suspense`와 호환되는 데이터 페칭 라이브러리를 사용해야 한다.
* **UI 파편화**: 너무 많은 `Suspense`를 중첩하면 화면 여기저기서 로딩 바가 따로 놀아 사용자 경험이 저해될 수 있다. 이를 해결하기 위해 `SuspenseList`(실험실 기능)나 상위 트리 설계가 중요하다.
