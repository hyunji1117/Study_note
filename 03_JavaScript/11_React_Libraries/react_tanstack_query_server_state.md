## 1. 개념 정의: 서버 상태 관리의 혁신

### **TanStack Query란?**
웹 애플리케이션에서 **서버 상태(Server State)**를 관리하기 위한 강력한 라이브러리입니다. 단순히 데이터를 가져오는 것을 넘어, 캐싱, 동기화, 업데이트 작업을 선언적으로 처리한다.

* **서버 상태(Server State)**: 클라이언트가 소유하지 않고 비동기 요청을 통해 가져와야 하는 외부의 데이터이다. 직접 제어할 수 없으므로 항상 최신 상태인지 확인하는 과정이 필요하다.

---

## 2. 사용 이유와 장단점 비교

### **주요 장점**
* **효율적 캐싱**: 동일 데이터에 대한 중복 요청을 방지하여 네트워크 비용을 줄이고 UX를 개선한다.
* **비동기 로직 단순화**: 데이터 fetching, refetching, invalidation 등을 선언적으로 관리하여 코드 가독성을 높힌다.
* **표준화된 상태 제공**: `isLoading`, `isError`, `data` 등 비동기 작업의 진행 상태를 직관적으로 제공한다.

### **단점 및 한계**
* **캐싱 전략의 복잡성**: `staleTime`, `gcTime` 등 세밀한 옵션 설정이 미흡할 경우 데이터 불일치가 발생할 수 있다.
* **학습 곡선**: Query Key 설계 및 캐시 무효화(Invalidation) 등 고유 개념 습득이 필요하다.
* **클라이언트 상태와의 분리**: 순수 UI 상태(모달 열림 등) 관리는 Redux나 Zustand 같은 별도 라이브러리가 필요할 수 있다.

## 3. 심화 개념 보강: 캐싱 메커니즘 (`staleTime` vs `gcTime`)

TanStack Query의 핵심은 **"데이터가 언제까지 신선(Fresh)한가?"**를 결정하는 기준이다.

| 옵션 | 의미 | 설명 |
| :--- | :--- | :--- |
| **staleTime** | 신선도 유지 시간 | 이 시간이 지나기 전까지는 캐시된 데이터를 'Fresh'하다고 간주하여 서버 요청을 생략한다. |
| **gcTime** | 가비지 컬렉션 시간 | 컴포넌트가 언마운트된 후, 캐시 데이터가 메모리에 남아있는 시간이다. 이후에는 삭제된다. |

> **💡 핵심 인사이트**
> TanStack Query는 **Zero-Config**를 지양하며 개발자가 데이터의 수명을 명시적으로 제어하게 한다. 특히 `useMutation`과 연계된 **Invalidation(무효화)** 전략은 서버 데이터를 변경한 즉시 관련 쿼리를 새로고침하여 데이터 무결성을 유지하는 최적의 방법이다.
